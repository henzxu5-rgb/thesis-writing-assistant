# 章节撰写

撰写论文的具体章节内容。严格按以下流程执行，确保每段论述有文献支撑。

## 工作流程（严格按顺序执行）

### Step 1: 确定写作范围

解析用户请求，确定：
- 要写的章节/小节编号或名称
- 写作模式：
  - **draft**（默认）：逐节撰写初稿。每次只写一个小节（如 2.1、3.2），写完后暂停，等待用户审阅反馈后再继续下一节。若用户指定了具体小节（如"写 2.1"），只写该节；若用户指定了章（如"写第二章"），从该章第一个未完成的小节开始。
  - **batch**：一次性生成完整章节初稿（仅在并行写作模式或用户明确要求"直接写完整章""一口气写完"时使用）
  - **expand**：在已有内容基础上扩展某一节
  - **revise**：根据用户反馈修改已有内容

### Step 2: 读取分章存放的大纲—写作计划文件

大纲采用分章存放，位于 `thesis/outline/` 目录下：
- `thesis/outline/index.md` — 全局索引（中心论点、推进逻辑、章节文件列表、参考文献缺口、写作状态）
- `thesis/outline/introduction.md` / `chapter1.md` / `chapter2.md` / `chapter3.md` / `conclusion.md` — 各章详细规划

**读取策略**：
1. 先读 `thesis/outline/index.md` 了解全局结构与目标章节在全文中的位置
2. 根据目标章节确定对应的分章文件：
   - 绪论 → `outline/introduction.md`
   - 第 N 章 → `outline/chapter<N>.md`
   - 结论 → `outline/conclusion.md`
3. 读取该分章文件，找到目标小节，提取四个关键字段：
   - **核心论点**：该节要论证什么
   - **论证思路**：论证如何展开（先 A 后 B 等）
   - **写法**：呈现方式与语气指示
   - **文献**：所需 chunk 列表（含每个 chunk 的关联说明）

**不要一次读取所有分章文件**——只读与目标章节相关的那一份。这是分章拆分的主要目的。

**同时读取用户偏好备忘**（若存在）：
- `thesis/writing-notes.md` — 用户持久化的风格/措辞/引用偏好（详见"上下文管理"节）

全局论证骨架、各章论证角色、概念依赖链、术语定型都已在 `outline/index.md` 中统一承载，不需要另设状态文件重复存储；前面各章的"隐性判断"已物化在 `chapters/chapter<N>.md` 正文里，AI 通过读取正文即可获取。

同时关注：
- 该章节在全文中的位置和作用（从 `index.md` 的推进逻辑中获取）
- 与前后章节的逻辑关系
- 该章节预计字数

如果 `thesis/outline/` 目录或对应分章文件不存在，或目标小节缺少上述四个字段中的任何一个，**停下并提示用户先运行 `/thesis-outline`** 完善规划，不要凭空补齐。

### Step 3: 加载参考文献

**直接从分章 outline 文件中拿到 chunk 列表**：
1. 从 Step 2 提取的"文献"字段中获取所需 chunk 路径
2. **直接并行读取**这些 chunk 文件（多个 Read 调用一次发出）
3. 跳过索引遍历，不需要读 `library/index.md` 或各书的 `index.md`
4. 如果该节在 `outline/index.md` 的"参考文献缺口"中被标记，告知用户文献不足

**禁止**：一次性读取所有小块文件或整个 library/ 目录；
**禁止**：在分章 outline 文件已列出 chunk 的情况下绕过它去自行检索。

### Step 4: 读取已有内容的末尾片段

**核心原则**：正文不承担全局记忆——全局论证链条、术语定型、各节核心论点都已由 `outline/index.md` 与 `outline/chapter<N>.md` 承担。读取正文**仅用于两项纯技术性需求**：语气衔接、避免重复直接引用。

脚注编号不是读正文的理由——每节/每章脚注独立编号即可，全文合并时统一重编。

**读取规则**：
- **本章已写部分 → 读末尾 ~500 字**（而非全文）：
  - 末尾段落用于语气/过渡衔接
  - 最近的直接引用用于避免重复
- **前一章 → 仅在开始新章的第一节时读末尾 2 段**：用于语气过渡
- **其他章节 → 一律不读**：跨章逻辑由 `outline/index.md` 的论证骨架承担，不加载正文

**禁止**：读取本章开头到中段的完整正文、读取其他章节正文、读取 `full_thesis.md` 或任何合并产物。

**如果需要呼应前面某节的具体论证**（如"如 2.1 所论证…"），从 `outline/chapter<N>.md` 中该节的"核心论点"字段读取——这才是 outline 的职能。若 outline 不够详细，应提示用户**补充 outline**，而不是让 AI 去正文里捞。

**draft 模式续写检测**：如果章节文件已存在且包含部分小节内容：
- 识别已完成的最后一个小节标题（`## X.Y`）
- 对照大纲确定下一个待写小节
- 向用户确认："已完成至 X.Y 节，接下来写 X.Z 节，是否继续？"

### Step 5: 撰写

**写作规范**：
- **章节编号格式**：统一使用阿拉伯数字编号，层级为 `1` → `1.1` → `1.1.1`。章标题用 `# 1　标题`，节标题用 `## 1.1　标题`，小节标题用 `### 1.1.1　标题`。不使用"第一章""一、""（一）"等中文序号。
- 论证结构：论点 → 论据（文献引用） → 分析 → 小结
- **脚注制**：每一处引用分配独立的脚注编号，文内用上标数字标注 `[1]`、`[2]`、`[3]`……编号按出现顺序递增，**同一文献在不同位置被引用时使用不同编号**。**每节/每章脚注独立编号**（都从 [1] 开始），不需要跨节或跨章续号——全文合并时统一重编
- 每段论述必须有文献支撑，引用来源必须是 library/ 中的实际内容
- **初稿优先直接引用**：在 draft 和 batch 模式撰写初稿时，引用文献内容应**优先采用直接引用**（加引号，从 chunk 逐字复制），而非用自己的话转述。直接引用便于用户在审阅阶段核对原文出处。将直接引用转化为更流畅的间接表述，留给后续 revise 模式处理。
- **直接引用必须逐字复制**：凡是加引号的直接引用（block quote 或行内引号），必须从 chunk 文件中**逐字逐句复制**原文，**严禁任何改写、概括或重组**。如需省略中间内容，用"……"标注省略位置。如果 chunk 原文有明显 OCR 错误，保留原文并用 `[sic]` 标注，或告知用户。间接引用（不加引号的转述）可以用自己的语言概括，但核心术语仍须与原文一致。
- **页码提取**：引用 chunk 内容时，找到所引文字最近的前置页码标记——格式为独占一行的 `[NNN]`（正则：`^\[(\d+)\]\s*$`）。将该数字作为脚注页码。如果所引内容跨两个标记（如 `[230]` 到 `[231]`），使用 `230-231`。如果 chunk 中无任何页码标记（常见于 EPUB 来源的英文文献），不标注页码——绝不可编造。
- 康德原文引用附德语原文，格式：`"中文引文"（"德语原文"）`
- **英文直接引用附中文翻译**：凡是直接引用英文文献原文的段落，在引文后紧跟括号附上中文翻译，格式：`"English quote"（中文翻译）`。短语级引用如果上下文已经清楚可酌情省略。
- 法律术语首次出现时给出定义或解释
- 段落之间有自然的逻辑过渡

**脚注格式（脚注制）**：

采用脚注制：每一处引用对应一条独立脚注，脚注使用 GB/T 7714 格式并标注具体页码。
引用信息从对应的 meta.md 或 chunk 文件的来源注释中获取。

**核心规则**：
- 每处引用 = 一个独立脚注编号 + 一条完整的引用条目（含页码）
- 同一文献被多次引用时，每次引用都是独立脚注，各标各的页码
- chunk 有页码标记 → 脚注末尾标注 `: 页码`（或 `: 起始页-结束页`）
- chunk 无页码标记 → 页码位置写 `[待补页码, 见 <chunk文件名>]`（如 `[待补页码, 见 chunk-03.md]`），绝不编造
- **不要**在正文中以 `(6:225)` 等形式行内标注科学院版页码，页码一律写在脚注中

示例（注意 [1] 和 [2] 引用同一本书的不同页码）：
```
正文：……它的简单性……当然一开始就必定使人惊奇。[1]
正文：……都被康德称为"自由的法则"[2]
正文：……应当就是《道德形而上学》这一体系[3]
正文：……将定言命令确立为唯一涉及道德法则的命令[4]
正文：……法权与强制之间存在分析性联系[5]

---
[1] 康德. 道德形而上学（注释本）[M]. 张荣, 李秋零, 译注. 北京: 中国人民大学出版社, 2013: 225.
[2] 康德. 道德形而上学（注释本）[M]. 张荣, 李秋零, 译注. 北京: 中国人民大学出版社, 2013: 214.
[3] 康德. 道德形而上学（注释本）[M]. 张荣, 李秋零, 译注. 北京: 中国人民大学出版社, 2013: 205.
[4] 康德. 道德形而上学的奠基[M]. 李秋零, 译. 北京: 中国人民大学出版社, 2013: 412-414.
[5] WILLASCHEK M. Right and Coercion[J]. International Journal of Philosophical Studies, 2009, 17(1): [待补页码, 见 chunk-03.md].
```

**保存**：将内容写入 `thesis/chapters/<章节文件名>.md`
文件命名与 `thesis/outline/` 下分章文件保持一一对应（`introduction.md` / `chapter1.md` / `chapter2.md` / `chapter3.md` / `conclusion.md`）。**不要**创建 `thesis/full_thesis.md` 或类似的全文合并文件——全文合并留到写作全部完成后的最终步骤。

### Step 5.5: 结构化报告与反馈循环（仅 draft 模式，batch 模式跳过此步）

写完当前小节后：

**1. 保存内容**：追加到 `thesis/chapters/<章节文件名>.md`

**2. 向用户提交结构化报告**，必须包含以下字段：

- **本节核心论点**：一句话复述本节实际完成的论证。若与 outline 预设有偏离，明确说明偏离点与理由。
- **字数**：约 X 字（对照 outline 预期）
- **使用文献**：列出每个 chunk 及引用页码
- **本节做出的隐性判断**（关键字段）：显式列出 AI 在写作中自行做出的、可能影响后续章节的判断。例如：
  - 对某个有歧义 chunk 的解读选择
  - 论证结构的偏好（先反驳再立论 vs. 先立论再辩护）
  - 术语译名或措辞的定型（"Rechtsstaat" 译为"法治国家"而非"法权国家"）
  - 对某个争议点的处理（回避 / 脚注简述 / 正文展开）
- **对前文的呼应点**：本节在哪些地方引用或承接了前面哪些节/章的结论
- **遗留给后续的钩子**：本节为后续哪一节埋下了伏笔或论证前提
- **下一节预告**：下一节标题 + outline 中的核心论点摘要
- **文献缺口提示**（如有）

**3. 停下，等待用户回复**。可能的回应：

- **"通过" / "继续"** → 直接开始写下一节（本节的隐性判断已随正文保存，无需额外状态文件）
- **修改意见** → 直接修改当前节，改完重新提交本节报告，循环直到用户通过
- **"跳过 X.Z"** → 跳到更后面的节（仍在 draft 模式）
- **持久化风格偏好** → 追加到 `thesis/writing-notes.md`；若涉及术语译名定型，则改写 `outline/index.md` 的"核心概念依赖链 → 术语定型"（详见"上下文管理"节）
- **"写完" / "一口气写完"** → 切换为 batch 模式完成剩余节（会失去过程把控，默认不建议）

**核心原则**：**未经用户明确通过不进入下一节**。隐性判断的显式披露是人机协同的关键——让 AI 的决策对用户可见、可纠正，而不是埋在正文中让用户事后才发现跑偏。

**隐性判断 → writing-notes.md 的联动**：披露隐性判断时，若其中某条判断具有**可复用性**（即未来其他节遇到类似情况也应当如此处理），AI 应在报告末尾主动询问："本节做出的判断 X 是否作为持久化风格规则写入 `writing-notes.md`？"例如"长引用统一用 block quote 而非行内引号"、"涉及凯尔斯汀德语原文时优先附中译"这类判断就适合固化为规则。用户确认后 AI 追加到 notes；拒绝则本节一次性处理，不入 notes。这样避免每次都要用户主动想起"啊这条要记下来"。

### Step 6: 自检

撰写完成后自动检查：
- 所有引用标注是否在脚注中有对应条目
- 引用的观点是否与 library/ 中小块文件的原文一致
- 论证逻辑是否与大纲一致
- 与前后章节的衔接是否自然
- 字数是否在大纲预计范围内

将自检结果简要报告给用户。

## 并行写作模式（应急模式，默认不推荐）

> ⚠️ **并行写作与本 skill 的人机协同理念相冲突。** 它一次派出多个子 Agent 同时写多章，用户无法逐节审阅、无法在早期纠正跑偏，子 Agent 内部的隐性判断完全不可见，事后章间衔接与术语统一的修订成本高。

**默认不主动建议此模式。** 当用户说"开始写"/"写论文"/"写所有章节"时，默认进入 **draft 模式**，从第一章第一节开始逐节写，不询问是否并行。

### 触发条件

**仅在以下任一条件满足时启动**：
- 用户**明确说出**"并行写"/"同时写多章"/"一口气出全部初稿"/"用子 agent 并行"
- 用户明确声明愿意承担事后大量衔接修订的代价

即使触发，也应先向用户重申代价（牺牲过程把控、需事后大量修订），确认后再执行。

### 执行流程

用户确认后：
1. 主上下文读取 `outline/index.md`，确定待写章节列表
2. 为每个章节生成独立 Agent 子任务，**并行启动**（同一条消息中发出多个 Agent 调用）
3. 每个子 agent 的 prompt 包含：
   - `thesis/outline/index.md` 路径（全局论证骨架，必读）
   - 对应分章 outline 路径（如 `thesis/outline/chapter2.md`）
   - `thesis/writing-notes.md` 路径（若存在，用户风格偏好）
   - 前一章末尾 2 段（衔接用，第一章省略）
   - 完整写作规范（Step 5 全部要求）
   - 保存路径：`thesis/chapters/<章节文件名>.md`
4. 主上下文等待所有子 agent 完成后：
   - 读取每章首尾各 2 段检查衔接
   - 检查脚注编号是否需要全文重编
   - 向用户报告完成情况与需人工审查的问题

**子 agent 规则**：每章 batch 模式一次写完，脚注从 [1] 开始独立编号。

**建议**：即使用户选择并行，绪论与结论也应在其他章节完成后**单独串行写**，因为它们需要呼应全文内容。

## 全文合并（最终步骤）

当用户明确要求"合并全文"/"拼全文"/"生成 full_thesis"时，按以下顺序把 `thesis/chapters/` 下的分章文件拼接为 `thesis/full_thesis.md`：

1. 按顺序拼接：`introduction.md` → `chapter1.md` → `chapter2.md` → `chapter3.md` → …（按章号）→ `conclusion.md`
2. **统一重编脚注**：写作阶段各节/各章脚注都是从 [1] 开始独立编号，合并时必须全文重新编号——按出现顺序从 [1] 递增，文内标注与脚注条目一一对应
3. 拼接正文内容保持原样，只修改脚注编号
4. 合并只在写作阶段基本完成后执行一次，日常写作/修改不要触发
5. 合并后可选：追加参考文献节（从各章脚注聚合去重）

**注意**：合并属于一次性动作，日常 `write-chapter` 流程**不**自动生成或更新 `full_thesis.md`。脚注重编也仅在合并时进行，日常写作每节独立编号即可。

## 上下文管理（draft 模式的长程写作支持）

draft 模式一节一停会产生大量累积对话，上下文窗口存在爆满风险。本 skill 采用"**outline 承担全部全局记忆 + 正文仅读末尾片段**"策略，让每一节的写作都可以从文件冷启动，不依赖对话历史，也不依赖正文全读。

### 核心设计原则

**outline 是 AI 唯一的全局信息源，正文只是 outline 的执行产物**。任何需要"全局定位"的信息（论证骨架、概念依赖、术语定型、各节核心论点、论证桥梁）都应在 `outline/index.md` 与 `outline/chapter<N>.md` 中可以找到。正文不承担全局记忆功能。

### 读取策略

- **全局定位靠 `outline/index.md`**：全文论证骨架（中心论点、各章论证角色、概念依赖链、关键论证桥梁、术语定型），写任何一节前必读。
- **本节规划靠 `outline/chapter<N>.md`**：目标小节的四字段（核心论点 / 论证思路 / 写法 / 文献）。
- **本章正文仅读末尾 ~500 字**：用于语气衔接、避免重复直接引用——**两项纯技术性需求**，不承担任何论证/逻辑功能。脚注编号不在此列（每节/每章独立编号，合并时重编）。
- **跨章衔接**：开始新章第一节时读前一章末尾 2 段（语气过渡），其余全由 index.md 承担。
- **其他章节正文一律不读**。

### Outline 的完备性要求

既然全局记忆完全由 outline 承担，outline 必须足够丰富：
- 若 AI 写某节时发现"我需要知道 X 节到底怎么论证的才能呼应"——说明 X 节在 `outline/chapter<N>.md` 的"核心论点"字段不够详细，**应补 outline**
- 若 AI 发现"某个术语前面怎么译的记不清了"——说明 `outline/index.md` 的"术语定型"不完整，**应补 index.md**
- 若 AI 发现"前一章结尾到这一章开头怎么衔接不清楚"——说明 `outline/index.md` 的"关键论证桥梁"缺失该处，**应补 index.md**

**原则**：当 AI 感到需要"去正文找信息"时，这是一个 outline 不完备的信号，应向用户报告并建议补 outline，而不是擅自加载正文全文。

### `thesis/writing-notes.md`（用户风格偏好备忘）

当用户给出**持久化的风格/措辞/引用偏好**时，AI 在用户通过本节后将其追加到此文件。新节开始时读取，用户不必反复说同一件事。

**适用范围（收紧）**：
- ✅ 措辞风格："避免'诚然''其实'等转折词"
- ✅ 引用方式："直接引用优先放在段落开头"、"凯尔斯汀的引用优先使用中译本"
- ✅ 段落结构偏好："每节开头先点明核心论点再展开论证"
- ❌ **术语译名不写这里**——术语定型属于论证骨架，应改写 `outline/index.md` 的"核心概念依赖链 → 术语定型"
- ❌ **一次性修改意见不写这里**——已经改到正文里了

格式：

```markdown
# 写作风格偏好备忘

- [引用] 直接引用优先放在段落开头
- [措辞] 避免"诚然""其实"等转折词
- [引用] 凯尔斯汀的引用优先使用中译本而非德语原文
- [结构] 每节开头先点明核心论点再展开论证
```

**识别时机**：用户的反馈若包含"以后都""不要再""统一用""记住"等关键词，或对某类事项的反复纠正，都应提取为持久化偏好写入此文件；**一次性的、仅针对本节的修改意见不写入**。

### 新节开始时的读取顺序（冷启动重建上下文）

1. `thesis/outline/index.md`（全局论证骨架，必读——各章论证角色、概念依赖链、论证桥梁、术语定型都在此）
2. `thesis/outline/chapter<N>.md` 本节条目（四字段：核心论点 / 论证思路 / 写法 / 文献）
3. `thesis/writing-notes.md`（用户风格偏好，全部，通常很短）
4. `thesis/chapters/chapter<N>.md` **末尾 ~500 字**（仅用于语气衔接与避免重复引用；若是新章第一节则改读前一章末尾 2 段）
5. 本节 outline 列出的 chunk 文件（并行读取）

这 5 项加起来每次约 2-4k token，远小于完整对话历史或正文全读。**即使当前对话完全清空、从新会话启动，读取以上文件后仍能无缝续写本节**——这是 draft 模式在长程写作中保持全局一致 + 章内连贯的根本机制。

**token 预算对照**（以 4500 字/章为例）：
- 原"本章全读"方案：每节约 5k token 正文加载
- 现"末尾片段"方案：每节约 0.8k token 正文加载
- 节省约 80%，随正文累积越到后面节省越显著

### `/clear` 与 `/compact` 时机建议

- **一节之内不要 /clear**：当前节的反馈循环需要完整对话上下文。
- **章与章之间是天然的 checkpoint**：一章完成后，所有持久化信息已在 `chapters/` 正文、`outline/index.md` 和 `writing-notes.md` 中，此时 `/clear` 无损失。
- **同一章多节之间**通常不需要 /clear；若对话明显臃肿可用 `/compact`。
- AI 可在每章末尾主动提示："本章已完成，可在此处 `/clear` 开始下一章。"

### 不依赖对话历史的保证

通过 `outline/index.md` + `chapters/chapter<N>.md` + `writing-notes.md` 的组合，任何一节的写作都满足：**即使当前对话完全清空，下次启动读取上述文件后仍能无缝续写**。对话历史只承担"当前节反馈循环"的临时载体作用，不承担长程记忆。

**三者的职能边界不重叠**：
- **`outline/index.md` + `outline/chapter<N>.md`** = 全局论证骨架 + 术语定型 + 各节核心论点（AI 唯一的全局信息源）
- **`chapters/chapter<N>.md`** = 正文产物（仅末尾片段被读取，用于语气/脚注/引用去重三项技术性需求）
- **`writing-notes.md`** = 风格规则（约束未来写作的偏好，不在正文中显现）

## 写作质量标准

- 论点明确，每段有中心句
- 论证充分，避免无依据的断言
- 学术用语规范，避免口语化
- 逻辑严密，避免跳跃式推理
- 引用准确，所有引文可追溯到文献库
