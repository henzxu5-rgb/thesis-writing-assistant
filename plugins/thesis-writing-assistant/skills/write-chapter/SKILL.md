# 章节撰写

撰写论文的具体章节内容。严格按以下流程执行，确保每段论述有文献支撑。

## 工作流程（严格按顺序执行）

### Step 1: 确定写作范围

解析用户请求，确定：
- 要写的章节/小节编号或名称
- 写作模式：
  - **draft**（默认）：逐节撰写初稿。每次只写一个小节（如 2.1、3.2），写完后暂停，等待用户审阅反馈后再继续下一节。若用户指定了具体小节（如"写 2.1"），只写该节；若用户指定了章（如"写第二章"），从该章第一个未完成的小节开始。
  - **batch**：一次性生成完整章节初稿（仅在并行写作模式或用户明确要求"直接写完整章""一口气写完"时使用）
  - **expand**：在已有内容基础上扩展某一节
  - **revise**：根据用户反馈修改已有内容

### Step 2: 读取分章存放的大纲—写作计划文件

大纲采用分章存放，位于 `thesis/outline/` 目录下：
- `thesis/outline/index.md` — 全局索引（中心论点、推进逻辑、章节文件列表、参考文献缺口、写作状态）
- `thesis/outline/introduction.md` / `chapter1.md` / `chapter2.md` / `chapter3.md` / `conclusion.md` — 各章详细规划

**读取策略**：
1. 先读 `thesis/outline/index.md` 了解全局结构与目标章节在全文中的位置
2. 根据目标章节确定对应的分章文件：
   - 绪论 → `outline/introduction.md`
   - 第 N 章 → `outline/chapter<N>.md`
   - 结论 → `outline/conclusion.md`
3. 读取该分章文件，找到目标小节，提取四个关键字段：
   - **核心论点**：该节要论证什么
   - **论证思路**：论证如何展开（先 A 后 B 等）
   - **写法**：呈现方式与语气指示
   - **文献**：所需 chunk 列表（含每个 chunk 的关联说明）

**不要一次读取所有分章文件**——只读与目标章节相关的那一份。这是分章拆分的主要目的。

**同时读取跨章记忆文件**（若存在）：
- `thesis/writing-state.md` — 前面各节通过后追加的浓缩状态（核心论点、术语/译名、钩子），用于跨章一致性
- `thesis/writing-notes.md` — 用户持久化偏好备忘（详见"上下文管理"节）

同时关注：
- 该章节在全文中的位置和作用（从 `index.md` 的推进逻辑中获取）
- 与前后章节的逻辑关系
- 该章节预计字数

如果 `thesis/outline/` 目录或对应分章文件不存在，或目标小节缺少上述四个字段中的任何一个，**停下并提示用户先运行 `/thesis-outline`** 完善规划，不要凭空补齐。

### Step 3: 加载参考文献

**直接从分章 outline 文件中拿到 chunk 列表**：
1. 从 Step 2 提取的"文献"字段中获取所需 chunk 路径
2. **直接并行读取**这些 chunk 文件（多个 Read 调用一次发出）
3. 跳过索引遍历，不需要读 `library/index.md` 或各书的 `index.md`
4. 如果该节在 `outline/index.md` 的"参考文献缺口"中被标记，告知用户文献不足

**禁止**：一次性读取所有小块文件或整个 library/ 目录；
**禁止**：在分章 outline 文件已列出 chunk 的情况下绕过它去自行检索。

### Step 4: 读取已有内容

正文按章拆分存放于 `thesis/chapters/`，每章一个文件（`introduction.md` / `chapter1.md` / `chapter2.md` / … / `conclusion.md`）。

**读取规则**：
- **本章已写部分 → 全读**：若 `thesis/chapters/<目标章节>.md` 已存在，**完整读取本章已写的全部内容**（不仅是末尾段落）。单章字数通常在 4500 字以内，成本低，完整读取是保证章内论证连贯的根本手段。
- **前一章 → 仅读末尾 2 段**：仅在开始新章的第一节时读取，用于语气/过渡衔接。
- **其他章节 → 不读**：跨章的全局一致性由 `writing-state.md` 承担，不加载其他章正文。

不要一次加载所有章节文件——按章拆分的目的正是避免这种上下文爆炸。

**draft 模式续写检测**：如果章节文件已存在且包含部分小节内容：
- 识别已完成的最后一个小节标题（`## X.Y`）
- 对照大纲确定下一个待写小节
- 向用户确认："已完成至 X.Y 节，接下来写 X.Z 节，是否继续？"

### Step 5: 撰写

**写作规范**：
- **章节编号格式**：统一使用阿拉伯数字编号，层级为 `1` → `1.1` → `1.1.1`。章标题用 `# 1　标题`，节标题用 `## 1.1　标题`，小节标题用 `### 1.1.1　标题`。不使用"第一章""一、""（一）"等中文序号。
- 论证结构：论点 → 论据（文献引用） → 分析 → 小结
- **脚注制**：每一处引用分配独立的脚注编号，文内用上标数字标注 `[1]`、`[2]`、`[3]`……编号按出现顺序递增，**同一文献在不同位置被引用时使用不同编号**
- 每段论述必须有文献支撑，引用来源必须是 library/ 中的实际内容
- **初稿优先直接引用**：在 draft 和 batch 模式撰写初稿时，引用文献内容应**优先采用直接引用**（加引号，从 chunk 逐字复制），而非用自己的话转述。直接引用便于用户在审阅阶段核对原文出处。将直接引用转化为更流畅的间接表述，留给后续 revise 模式处理。
- **直接引用必须逐字复制**：凡是加引号的直接引用（block quote 或行内引号），必须从 chunk 文件中**逐字逐句复制**原文，**严禁任何改写、概括或重组**。如需省略中间内容，用"……"标注省略位置。如果 chunk 原文有明显 OCR 错误，保留原文并用 `[sic]` 标注，或告知用户。间接引用（不加引号的转述）可以用自己的语言概括，但核心术语仍须与原文一致。
- **页码提取**：引用 chunk 内容时，找到所引文字最近的前置页码标记——格式为独占一行的 `[NNN]`（正则：`^\[(\d+)\]\s*$`）。将该数字作为脚注页码。如果所引内容跨两个标记（如 `[230]` 到 `[231]`），使用 `230-231`。如果 chunk 中无任何页码标记（常见于 EPUB 来源的英文文献），不标注页码——绝不可编造。
- 康德原文引用附德语原文，格式：`"中文引文"（"德语原文"）`
- **英文直接引用附中文翻译**：凡是直接引用英文文献原文的段落，在引文后紧跟括号附上中文翻译，格式：`"English quote"（中文翻译）`。短语级引用如果上下文已经清楚可酌情省略。
- 法律术语首次出现时给出定义或解释
- 段落之间有自然的逻辑过渡

**脚注格式（脚注制）**：

采用脚注制：每一处引用对应一条独立脚注，脚注使用 GB/T 7714 格式并标注具体页码。
引用信息从对应的 meta.md 或 chunk 文件的来源注释中获取。

**核心规则**：
- 每处引用 = 一个独立脚注编号 + 一条完整的引用条目（含页码）
- 同一文献被多次引用时，每次引用都是独立脚注，各标各的页码
- chunk 有页码标记 → 脚注末尾标注 `: 页码`（或 `: 起始页-结束页`）
- chunk 无页码标记 → 页码位置写 `[待补页码, 见 <chunk文件名>]`（如 `[待补页码, 见 chunk-03.md]`），绝不编造
- **不要**在正文中以 `(6:225)` 等形式行内标注科学院版页码，页码一律写在脚注中

示例（注意 [1] 和 [2] 引用同一本书的不同页码）：
```
正文：……它的简单性……当然一开始就必定使人惊奇。[1]
正文：……都被康德称为"自由的法则"[2]
正文：……应当就是《道德形而上学》这一体系[3]
正文：……将定言命令确立为唯一涉及道德法则的命令[4]
正文：……法权与强制之间存在分析性联系[5]

---
[1] 康德. 道德形而上学（注释本）[M]. 张荣, 李秋零, 译注. 北京: 中国人民大学出版社, 2013: 225.
[2] 康德. 道德形而上学（注释本）[M]. 张荣, 李秋零, 译注. 北京: 中国人民大学出版社, 2013: 214.
[3] 康德. 道德形而上学（注释本）[M]. 张荣, 李秋零, 译注. 北京: 中国人民大学出版社, 2013: 205.
[4] 康德. 道德形而上学的奠基[M]. 李秋零, 译. 北京: 中国人民大学出版社, 2013: 412-414.
[5] WILLASCHEK M. Right and Coercion[J]. International Journal of Philosophical Studies, 2009, 17(1): [待补页码, 见 chunk-03.md].
```

**保存**：将内容写入 `thesis/chapters/<章节文件名>.md`
文件命名与 `thesis/outline/` 下分章文件保持一一对应（`introduction.md` / `chapter1.md` / `chapter2.md` / `chapter3.md` / `conclusion.md`）。**不要**创建 `thesis/full_thesis.md` 或类似的全文合并文件——全文合并留到写作全部完成后的最终步骤。

### Step 5.5: 结构化报告与反馈循环（仅 draft 模式，batch 模式跳过此步）

写完当前小节后：

**1. 保存内容**：追加到 `thesis/chapters/<章节文件名>.md`

**2. 向用户提交结构化报告**，必须包含以下字段：

- **本节核心论点**：一句话复述本节实际完成的论证。若与 outline 预设有偏离，明确说明偏离点与理由。
- **字数**：约 X 字（对照 outline 预期）
- **使用文献**：列出每个 chunk 及引用页码
- **本节做出的隐性判断**（关键字段）：显式列出 AI 在写作中自行做出的、可能影响后续章节的判断。例如：
  - 对某个有歧义 chunk 的解读选择
  - 论证结构的偏好（先反驳再立论 vs. 先立论再辩护）
  - 术语译名或措辞的定型（"Rechtsstaat" 译为"法治国家"而非"法权国家"）
  - 对某个争议点的处理（回避 / 脚注简述 / 正文展开）
- **对前文的呼应点**：本节在哪些地方引用或承接了前面哪些节/章的结论
- **遗留给后续的钩子**：本节为后续哪一节埋下了伏笔或论证前提
- **下一节预告**：下一节标题 + outline 中的核心论点摘要
- **文献缺口提示**（如有）

**3. 停下，等待用户回复**。可能的回应：

- **"通过" / "继续"** → 先更新 `thesis/writing-state.md`（追加本节浓缩条目），然后开始写下一节
- **修改意见** → 直接修改当前节，改完重新提交本节报告，循环直到用户通过
- **"跳过 X.Z"** → 跳到更后面的节（仍在 draft 模式）
- **持久化偏好** → 追加到 `thesis/writing-notes.md`（详见"上下文管理"节）
- **"写完" / "一口气写完"** → 切换为 batch 模式完成剩余节（会失去过程把控，默认不建议）

**核心原则**：**未经用户明确通过不进入下一节**。隐性判断的显式披露是人机协同的关键——让 AI 的决策对用户可见、可纠正，而不是埋在正文中让用户事后才发现跑偏。

### Step 6: 自检

撰写完成后自动检查：
- 所有引用标注是否在脚注中有对应条目
- 引用的观点是否与 library/ 中小块文件的原文一致
- 论证逻辑是否与大纲一致
- 与前后章节的衔接是否自然
- 字数是否在大纲预计范围内

将自检结果简要报告给用户。

## 并行写作模式（应急模式，默认不推荐）

> ⚠️ **并行写作与本 skill 的人机协同理念相冲突。** 它一次派出多个子 Agent 同时写多章，用户无法逐节审阅、无法在早期纠正跑偏，子 Agent 内部的隐性判断完全不可见，事后章间衔接与术语统一的修订成本高。

**默认不主动建议此模式。** 当用户说"开始写"/"写论文"/"写所有章节"时，默认进入 **draft 模式**，从第一章第一节开始逐节写，不询问是否并行。

### 触发条件

**仅在以下任一条件满足时启动**：
- 用户**明确说出**"并行写"/"同时写多章"/"一口气出全部初稿"/"用子 agent 并行"
- 用户明确声明愿意承担事后大量衔接修订的代价

即使触发，也应先向用户重申代价（牺牲过程把控、需事后大量修订），确认后再执行。

### 执行流程

用户确认后：
1. 主上下文读取 `outline/index.md`，确定待写章节列表
2. 为每个章节生成独立 Agent 子任务，**并行启动**（同一条消息中发出多个 Agent 调用）
3. 每个子 agent 的 prompt 包含：
   - 对应分章 outline 路径（如 `thesis/outline/chapter2.md`）
   - `thesis/writing-state.md` 路径（若存在，用于跨章一致性参考）
   - `thesis/writing-notes.md` 路径（若存在，用户偏好备忘）
   - 前一章末尾 2 段（衔接用，第一章省略）
   - 完整写作规范（Step 5 全部要求）
   - 保存路径：`thesis/chapters/<章节文件名>.md`
4. 主上下文等待所有子 agent 完成后：
   - 读取每章首尾各 2 段检查衔接
   - 检查脚注编号是否需要全文重编
   - 向用户报告完成情况与需人工审查的问题

**子 agent 规则**：每章 batch 模式一次写完，脚注从 [1] 开始独立编号。

**建议**：即使用户选择并行，绪论与结论也应在其他章节完成后**单独串行写**，因为它们需要呼应全文内容。

## 全文合并（最终步骤）

当用户明确要求"合并全文"/"拼全文"/"生成 full_thesis"时，按以下顺序把 `thesis/chapters/` 下的分章文件拼接为 `thesis/full_thesis.md`：

1. `introduction.md` → `chapter1.md` → `chapter2.md` → `chapter3.md` → …（按章号）→ `conclusion.md`
2. 拼接时保留各章原样，不改内容；如脚注采用每章独立编号而需要全文连续编号，先征求用户意见
3. 合并只在写作阶段基本完成后执行一次，日常写作/修改不要触发
4. 合并后可选：追加参考文献节（从各章脚注聚合去重）

**注意**：合并属于一次性动作，日常 `write-chapter` 流程**不**自动生成或更新 `full_thesis.md`。

## 上下文管理（draft 模式的长程写作支持）

draft 模式一节一停会产生大量累积对话，上下文窗口存在爆满风险。本 skill 采用"**章内全读 + 跨章状态文件**"策略，让每一节的写作都可以从文件冷启动，不依赖对话历史。

### 核心策略：章内全读 + 跨章摘要

- **同一章内续写**：每次开始新一节前，**完整读取 `thesis/chapters/chapter<N>.md` 已写部分**。单章字数通常在 3500-4500 字以内，读取成本低，能完整保证章内论证连贯性。
- **跨章时**：只读 `thesis/writing-state.md` 拿到前面各章的浓缩状态 + 前一章末尾 2 段（语气衔接），**不读前面章节全文**。

### `thesis/writing-state.md`（跨章记忆文件）

每完成一节且用户通过后，AI 必须更新此文件，追加本节条目。若首次创建，使用以下骨架：

```markdown
# 写作状态（AI 跨章记忆）

> 本文件由 write-chapter 在每节写作通过后自动维护，用于新节/新章开始时重建上下文，避免依赖对话历史。

## 第 X 章　<章标题>

### X.Y　<节标题>
- **核心论点**：<一句话>
- **关键术语/译名**：<本节引入或定型的>
- **对后续的钩子**：<为哪节埋下伏笔>
- **遗留问题**：<若有未解决、需后续处理的点>
```

条目应当**浓缩**——一节对应 4-6 行即可，不要复制正文段落。

### `thesis/writing-notes.md`（用户偏好备忘）

当用户给出**持久化的偏好或纠正**（例如"'法权'不译为'权利'"、"直接引用优先放在段落开头"、"避免'诚然''其实'这类转折词"），AI 在用户通过本节后将其追加到此文件。新节开始时读取，用户不必反复说同一件事。

格式：

```markdown
# 写作偏好备忘

- [术语] 'Rechtsstaat' 统一译为"法治国家"
- [引用] 直接引用优先放在段落开头
- [措辞] 避免"诚然""其实"等转折词
- [论证] 凯尔斯汀的引用优先使用中译本而非德语原文
```

**识别时机**：用户的反馈若包含"以后都""不要再""统一用""记住"等关键词，或对某类事项的反复纠正，都应提取为持久化偏好写入此文件；**一次性的、仅针对本节的修改意见不写入**。

### 新节开始时的读取顺序（冷启动重建上下文）

1. `thesis/outline/index.md`（全局论点与推进逻辑，简短）
2. `thesis/outline/chapter<N>.md` 本节条目（四字段）
3. `thesis/writing-state.md`（全部，通常很短）
4. `thesis/writing-notes.md`（用户偏好，全部）
5. `thesis/chapters/chapter<N>.md` 本章已写部分（全读；若是新章第一节则读前一章末尾 2 段）
6. 本节 outline 列出的 chunk 文件（并行读取）

这 6 项加起来通常远小于完整对话历史。**即使当前对话完全清空、从新会话启动，读取以上文件后仍能无缝续写本节**——这是 draft 模式在长程写作中保持章内连贯 + 全局一致的根本机制。

### `/clear` 与 `/compact` 时机建议

- **一节之内不要 /clear**：当前节的反馈循环需要完整对话上下文（用户的修改意见、AI 的响应）。
- **章与章之间是天然的 checkpoint**：一章完成后 `writing-state.md` 已更新完毕，此时 `/clear` 无损失，新章冷启动即可。
- **同一章多节之间**通常不需要 /clear；若对话明显臃肿可用 `/compact`。
- AI 可在每章末尾主动提示："本章已完成，`writing-state.md` 已更新，可在此处 `/clear` 开始下一章。"

### 不依赖对话历史的保证

通过 `writing-state.md` + `writing-notes.md` + `chapters/chapter<N>.md` 的组合，任何一节的写作都满足：**即使当前对话完全清空，下次启动读取上述文件后仍能无缝续写**。对话历史只承担"当前节反馈循环"的临时载体作用，不承担长程记忆。

## 写作质量标准

- 论点明确，每段有中心句
- 论证充分，避免无依据的断言
- 学术用语规范，避免口语化
- 逻辑严密，避免跳跃式推理
- 引用准确，所有引文可追溯到文献库
